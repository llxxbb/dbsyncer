## 背景

当前字段差异分析比较的是**源表字段 vs 目标表字段**，涵盖 4 种差异类型：
- `ADDED`：目标表有但源表没有
- `MISSING`：源表有但目标表没有
- `TYPE_MISMATCH`：类型不匹配
- `LENGTH_MISMATCH`：长度不匹配

**问题场景：**
mapping 配置了 target 字段，但目标表实际不存在该字段。当前差异分析无法发现此问题，导致数据同步时可能静默失败或跳过，运维也无法及时发现配置与实际表结构的偏差。

## 决策

在字段差异分析中增加第 5 种差异类型：**`MAPPING_ONLY`**。

**核心策略：**
1. **独立检测**：不将此类差异与现有 `MISSING` 或 `ADDED` 混淆，保持语义清晰。
2. **配置清理**：针对映射目标字段不存在的情况，提供清理无效 Mapping 配置的能力。

## 影响

- **技术影响**：后端新增 `MAPPING_ONLY` 检测逻辑，扩展 `getFieldDifference()` 用于差异检测；修复预览通过独立接口按需计算（用户点击修复后触发），避免全量差异计算时的冗余开销；前端差异弹窗与修复弹窗增加对应区块与勾选逻辑。

## 验收标准

1. **检测准确**：mapping 配置了 target 字段，目标表实际无此字段 → 弹窗显示 `MAPPING_ONLY`。
2. **修复 - 移除**：选中移除操作后，fieldMapping 中对应的配置被删除，无 SQL 执行。
3. **交互**：支持多选批量清理，也支持单条清理。
4. **兼容性**：不影响现有 4 种差异类型的检测与修复流程。

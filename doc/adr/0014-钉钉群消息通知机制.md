## 背景

同步任务出现异常时（任务异常中断、数据写入错误队列等），运维人员无法第一时间获知。当前通知系统仅支持邮件渠道（`MailNotifyServiceProvider`），且存在以下问题：

1. **单渠道限制**：`NotifyService` 为单例 bean，邮件是唯一的实现，无法灵活组合多渠道
2. **缺少事件类型区分**：`NotifyMessage` 只有 title/content/receivers，无法按事件类型过滤通知
3. **钉钉场景缺失**：团队日常沟通使用钉钉，群消息通知是刚需

现有触发点分布：
- `MonitorServiceImpl` → 同步失败
- `MappingServiceImpl` → 手动停止/重置驱动
- `TableGroupServiceImpl` → 手动重新同步

**缺失的触发点**：数据写入错误队列时（`MetaBufferActuator` 中 `flushStrategy.flushIncrementData()` 调用），目前只有日志记录，没有通知。

## 决策

**新增钉钉 Webhook 机器人通知渠道，重构通知系统支持多通道组合和事件类型过滤。**

### 核心变更

#### 1. 通知渠道重构：组合模式

引入 `CompositeNotifyService`，聚合多个 `NotifyService` 实现：

```
NotifyService (接口)
├── CompositeNotifyService (新增) — 聚合所有已启用的通道
├── MailNotifyServiceProvider (现有) — 邮件通知
└── DingTalkNotifyServiceProvider (新增) — 钉钉 Webhook
```

- `NotifySupportConfiguration` 改为 `CompositeNotifyService` 作为默认 bean
- 通过全局 `channels` 列表控制启用哪些通道，替代各通道内的 `enabled` 字段
- 组合服务遍历所有已启用的通道，逐一调用 `sendMessage()`

#### 2. 事件类型枚举

在 `NotifyMessage` 中增加 `type` 字段：

```java
public enum NotifyType {
    TASK_INTERRUPTED,   // 任务异常中断
    DATA_ERROR_QUEUE,   // 数据进入错误队列
    MANUAL_OPERATION    // 手动操作（停止/重置/重新同步等）
}
```

#### 3. 钉钉 Webhook 配置

使用钉钉自定义机器人（Webhook 方式），配置前缀 `dbsyncer.plugin.notify.dingtalk.*`：

```yaml
dbsyncer:
  plugin:
    notify:
      enabled: true
      channels:                     # 控制启用哪些通道
        - mail
        - dingtalk
      mail:
        username: "xxx@qq.com"
        password: "***"
      dingtalk:
        webhook-url: "https://oapi.dingtalk.com/robot/send?access_token=xxx"
        secret: "***"               # 必选，加签密钥
        keywords: "dbsyncer 警报"    # 缺省值
        notify-types:               # 可选，默认全部通知
          - TASK_INTERRUPTED
          - DATA_ERROR_QUEUE
```

- **Webhook URL**：钉钉群机器人 Webhook 地址
- **Secret**：加签密钥（必选，启用加签验证）
- **keywords**：缺省值 `dbsyncer 警报`，消息必须包含该关键词才能发送（钉钉侧限制）
- **notify-types**：可选，白名单机制，只通知指定的事件类型
- **receivers 字段处理**：钉钉通道忽略该字段（群消息全员可见）

#### 4. 补充错误队列通知触发点

在 `FlushStrategyImpl.flushIncrementData()` 中，当结果包含错误数据时（`result.getFailData()` 非空且 `result.error` 非空），调用通知服务发送 `DATA_ERROR_QUEUE` 类型消息。

### 影响

- **技术影响**：
  - `NotifyService` 接口不变，向后兼容
  - `NotifyMessage` 增加 `type` 字段，现有调用不受影响
  - `CompositeNotifyService` 作为新的默认通知服务，替代空实现
  - 新增 `DingTalkNotifyServiceProvider` 实现
  - 新增 `NotifyType` 枚举
- **运维影响**：
  - 钉钉群可实时收到同步异常告警
  - 可按事件类型灵活配置通知通道
  - 多渠道（邮件+钉钉）可同时启用
- **开发影响**：
  - 新增通知触发点需要指定 `NotifyType`
  - 现有通知调用保持不变（向后兼容）

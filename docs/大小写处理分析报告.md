## 1. 概览

项目对**字段类型名**的大小写处理是统一的（全部转为大写），但对**字段名、表名、Schema 名**的大小写处理**存在显著不一致**。

| 维度 | 处理策略 | 一致性 |
|------|---------|--------|
| 字段类型名 (typeName) | 统一 `toUpperCase()` | ✅ 一致 |
| 字段名匹配 (field mapping) | 部分 `toLowerCase()`, 部分原始大小写 | ❌ 不一致 |
| 表名匹配 (table mapping) | 部分 `equalsIgnoreCase()`, 部分原始大小写 | ❌ 不一致 |
| Schema 名 | Oracle 强制大写，PG 转小写，SQL Server 无处理 | ❌ 各 connector 策略不同 |
| 主键匹配 | 部分 `equalsIgnoreCase()`, 部分 `contains()` | ❌ 不一致 |

## 3. Schema 名处理差异

### 3.1 Oracle — 强制大写

| 文件 | 行号 | 代码 |
|------|------|------|
| `OracleConfigValidator.java` | 26 | `schema.toUpperCase()` |
| `DqlOracleConfigValidator.java` | 25 | `schema.toUpperCase()` |
| `OracleConnector.java` | 134 | `effectiveSchema.toUpperCase()` |
| `OracleTemplate.java` | 287-292 | `UPPER('%s') AND table_name = UPPER('%s')` |
| `LogMinerHelper.java` | 151 | `logMinerUser.toUpperCase()` |
| `LogMinerHelper.java` | 202 | `list.add(k.toUpperCase())` |

**评价**：Oracle 全部大写是合理的，因为 Oracle 默认标识符就是大写的。

### 3.2 PostgreSQL — 部分小写

| 文件 | 行号 | 代码 |
|------|------|------|
| `PgOutputMessageDecoder.java` | 99 | `String.format("dbs_pub_%s_%s", ...).toLowerCase()` |
| `AbstractMessageDecoder.java` | 54 | `String.format("dbs_slot_%s_%s_%s", ...).toLowerCase()` |
| `PostgreSQLTemplate.java` | 153 | `ddlType.equalsIgnoreCase("INT")` |

**评价**：Publication 和 Slot 名统一转小写，但 Schema/Table/Field 名未做统一处理。PostgreSQL 默认将未引号标识符转为小写，但引号标识符保留原始大小写。

### 3.3 SQL Server — 无统一处理

| 文件 | 行号 | 代码 |
|------|------|------|
| `SqlServerCTListener.java` | 230 | `schema = cfg.getSchema(); if empty → "dbo"` |
| `SqlServerListener.java` | 209 | `schema.equals(schemaName)` — 区分大小写 |

**评价**：SQL Server 的默认 collation（通常 `SQL_Latin1_General_CP1_CI_AS`）是不区分大小写的，但代码层面没有统一处理。

### 3.4 MySQL — 依赖操作系统

MySQL 的表名大小写敏感程度取决于 `lower_case_table_names` 系统变量：
- Linux 默认 = 0（区分大小写）
- Windows 默认 = 1（不区分，存储小写）
- macOS 默认 = 2（不区分，存储原始大小写）

项目对此没有统一处理。

## 7. 统一建议

### 7.1 核心建议：提取统一的大小写处理工具类

建议创建 `CaseInsensitiveNameUtil` 工具类：

```java
public final class CaseInsensitiveNameUtil {
    
    /** 将字段名/表名/Schema名统一转为小写（用于 Map key 匹配） */
    public static String toLowerKey(String name) {
        return name == null ? null : name.toLowerCase();
    }
    
    /** 构建不区分大小写的字段 Map */
    public static Map<String, Field> toFieldMap(List<Field> fields) {
        Map<String, Field> map = new HashMap<>();
        if (fields != null) {
            fields.forEach(f -> map.put(toLowerKey(f.getName()), f));
        }
        return map;
    }
    
    /** 不区分大小写的字段名比较 */
    public static boolean nameEquals(String name1, String name2) {
        return StringUtil.equalsIgnoreCase(name1, name2);
    }
    
    /** 不区分大小写的 List.contains */
    public static boolean containsIgnoreCase(List<String> list, String name) {
        return list != null && list.stream().anyMatch(n -> StringUtil.equalsIgnoreCase(n, name));
    }
}
```

### 7.3 关于 Schema 名的建议

- **Oracle**：保持当前 `toUpperCase()` 策略（与 Oracle 默认行为一致）
- **PostgreSQL**：建议对 Schema/Table 名不做强制转换，依赖 `buildTable()/buildColumn()` 的引号处理
- **SQL Server**：建议保持一致的 collation 感知处理
- **MySQL**：建议在配置层面增加 `lower_case_table_names` 的检测或提示

### 7.4 关于 DDL 生成的建议

当前 `buildTable()/buildColumn()` 直接使用传入的 name + 引号，不改变大小写。这是**正确的**——DDL 应该保持原始大小写，由数据库自身 collation 决定行为。大小写统一只在**匹配/查找**层做，不在**生成**层做。

## 8. 总结

| 评估维度 | 结论 |
|---------|------|
| 字段类型名 | ✅ 统一大写，无风险 |
| 字段差异对比 | ✅ 统一小写 key，无风险 |
| DDL 变更映射 | ❌ 区分大小写，高风险 |
| 主键匹配 | ❌ 部分区分大小写，高风险 |
| Schema 名 | ⚠️ 各 connector 策略不同，需评估 |
| 是否需要统一工具类 | ✅ 强烈建议 |

**最关键的风险**：当源数据库（如 MySQL on Linux，区分大小写）与目标数据库（如 PostgreSQL，默认小写）之间存在大小写差异的字段名/表名时，DDL 解析和主键匹配可能失败，导致字段映射错误、主键丢失标记等问题。
